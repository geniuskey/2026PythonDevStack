# 프롤로그: 2026년의 파이썬 생태계

## 세 가지 질문

만약 당신이 지금 파이썬 프로젝트를 시작한다면:

1. **패키지 관리는 뭘로 하지?** `pip`, `pipenv`, `poetry`, `pdm`, `uv`...
2. **웹 프레임워크는?** `Django`, `Flask`, `FastAPI`...
3. **코드 품질 도구는?** `flake8`, `black`, `pylint`, `ruff`...

이 질문들 앞에서 멈춰 본 적 있나요?

**이것이 바로 이 책이 존재하는 이유입니다.**

---

## 풍요 속의 혼란

파이썬 생태계는 놀라울 만큼 풍부합니다. PyPI에는 50만 개가 넘는 패키지가 있고, 어떤 문제든 해결할 라이브러리가 존재합니다.

하지만 이 풍요로움은 동시에 **선택의 고통**을 만들어냅니다.

- 같은 일을 하는 라이브러리가 10개씩 있고
- 누군가는 A를 추천하고, 다른 누군가는 B가 더 낫다고 합니다
- 3년 전 튜토리얼은 C를 쓰라고 하지만, 지금도 유효한지 모르겠습니다

**"지금 시작하면 뭘 써야 하지?"**

이 단순한 질문에 답하기가 점점 어려워지고 있습니다.

---

## 이 책이 제공하는 것

### 1. 지도 (Map)

파이썬으로 할 수 있는 모든 것의 **전체 그림**:

- 웹 개발부터 데이터 과학까지
- 시스템 자동화부터 AI 애플리케이션까지
- 문서 처리부터 멀티미디어까지

**압도적인 커버리지**로 "이런 것도 파이썬으로 할 수 있구나"를 발견하게 합니다.

### 2. 나침반 (Compass)

하지만 이 책은 단순한 라이브러리 목록이 아닙니다. 각 라이브러리마다:

- ⭐ **2026 추천**: 지금 시작한다면 이것
- ⚠️ **레거시/유지보수용**: 기존 코드에서만
- 🔄 **대체재 존재**: 더 나은 선택지 있음
- 🧩 **특정 상황 특화**: 특수 목적용

**명확한 신호**를 통해 당신이 판단할 수 있게 돕습니다.

### 3. 가이드 (Guide)

"라이브러리 A가 좋다"는 것만으로는 부족합니다. 실제로는:

- **API 서버를 만들려면?** → FastAPI + pydantic + httpx + ...
- **패키지를 배포하려면?** → pyproject.toml + uv + build + ...
- **LLM 서비스를 운영하려면?** → langchain + vector DB + tracing + ...

**문제 해결 흐름**과 **라이브러리 조합**을 함께 제시합니다.

## 2026년이라는 시점

이 책은 2023년에 시작되었지만, **2026년을 기준으로 전면 개정**되었습니다.

### 지난 3년간 일어난 일들

**패키징 혁명**
- `setup.py` → `pyproject.toml`
- `pipenv` 시대 → `uv` 등장 (10-100배 속도)

**코드 품질 도구의 통합**
- `flake8 + black + isort` 조합 → `ruff` 하나로 (50배 속도)

**웹 개발의 패러다임 전환**
- `Flask` 중심 → `FastAPI` 표준화
- 동기 I/O → 비동기 기본

**데이터 처리의 진화**
- `pandas` 독주 → `polars` 부상 (10배 빠름)
- SQL 파일 → `duckdb` (인메모리 분석)

**LLM 시대의 도래**
- 실험적 도구 → 프로덕션 스택
- RAG, eval, tracing이 필수 개념으로

### 하지만 변하지 않는 것들

동시에 여전히 강력한 것들도 있습니다:

- `requests`는 여전히 간결하고 강력합니다
- `Django`는 여전히 전체 솔루션으로 최고입니다
- `pandas`는 여전히 데이터 과학의 중심입니다
- `pytest`는 여전히 테스팅의 표준입니다

**"레거시"라고 해서 나쁜 것이 아닙니다.**
상황에 따라 여전히 최선의 선택일 수 있습니다.

## 이 책이 하지 않는 것

### ❌ 튜토리얼이 아닙니다

각 라이브러리의 사용법을 단계별로 가르치지 않습니다. 대신:

- **무엇을 선택해야 하는지**
- **왜 그것이 2026년 기준인지**
- **어떤 조합으로 사용하는지**

를 알려줍니다.

### ❌ 완전한 레퍼런스가 아닙니다

각 라이브러리의 모든 API를 다루지 않습니다. 공식 문서가 더 낫습니다.

### ❌ 절대적 진리가 아닙니다

"FastAPI가 Flask보다 무조건 낫다"고 말하지 않습니다. 대신:

> "2026년 신규 API 프로젝트라면 FastAPI를 추천. 하지만 Django 생태계 안에 있거나, 간단한 도구라면 Flask도 여전히 훌륭함"

처럼 **맥락**과 함께 제시합니다.

## 생태계는 살아있다

이 책이 출판되는 순간에도 파이썬 생태계는 진화합니다:

- 새로운 라이브러리가 나타나고
- 기존 도구들이 더 빨라지고
- 패러다임이 조금씩 바뀝니다

**이 책은 "2026년 1월의 스냅샷"입니다.**

하지만 여기서 제시하는 **선택의 기준**과 **판단의 프레임워크**는 더 오래 유효할 것입니다:

1. **생태계의 momentum**: 커뮤니티가 어디로 움직이는가?
2. **성숙도**: 프로덕션에 쓸 만큼 안정적인가?
3. **현대성**: 현대 파이썬 패턴을 따르는가?
4. **조합성**: 다른 도구들과 잘 어울리는가?
